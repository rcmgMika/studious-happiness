📋
Criterion 01
Description

Euler's number e is a special mathematical constant, approximately equal to 2.71828182845…. It is irrational (never-ending, non-repeating decimal) and transcendental (not the root of any polynomial with rational coefficients). It is defined as:

e = lim(n→∞) (1 + 1/n)ⁿ ← limit definition e = 1 + 1/1! + 1/2! + 1/3! + 1/4! + ... ← series definition = 1 + 1 + 0.5 + 0.1667 + 0.0417 + ... ≈ 2.71828182845904523536... -- e is irrational: it never terminates or repeats -- e appears naturally wherever continuous growth or decay exists

The natural logarithm, written ln(x), is the logarithm with base e. It is the inverse of the exponential function eˣ:

ln(x) = log_e(x) = n means eⁿ = x Examples: ln(e) = 1 because e¹ = e ln(e³) = 3 because e³ = e³ ln(1) = 0 because e⁰ = 1 ln(e⁻²) = −2 because e⁻² = e⁻² ln(7.389) ≈ 2 because e² ≈ 7.389 Inverse relationship: e^(ln x) = x and ln(eˣ) = x ← they perfectly undo each other

Why is e the "natural" base? Unlike base 10 (chosen for human convenience of counting), e arises naturally from the mathematics of continuous change. It is the unique number where the function eˣ is its own derivative — meaning its rate of change at every point equals its own value. No other base has this property.

d/dx [eˣ] = eˣ ← the exponential function is its own derivative d/dx [ln x] = 1/x ← the derivative of the natural log -- These two facts make e and ln indispensable in calculus and optimization

All logarithm laws from the previous subtopic apply fully to ln, just with base e implicit:

ln(xy) = ln(x) + ln(y) ← product law ln(x/y) = ln(x) − ln(y) ← quotient law ln(xⁿ) = n · ln(x) ← power law (most used in ML) ln(e) = 1 ← identity ln(1) = 0 ← identity

Criterion 02
Purpose
CalculusThe derivative of ln(x) = 1/x and the derivative of eˣ = eˣ are the two most frequently used results in calculus. Every optimization problem in machine learning ultimately uses these. Without ln and e, calculus-based optimization cannot be done.
ML LossCross-entropy loss — the backbone loss function for all classification problems — is defined using ln: L = −Σ yᵢ · ln(ŷᵢ). The power law of ln is what makes this function differentiable and optimizable via gradient descent.
LogisticThe sigmoid activation function σ(x) = 1/(1 + e⁻ˣ) uses e directly. Logistic regression — the fundamental classification model — is built entirely on eˣ and ln. Log-odds = ln(p / (1−p)).
StatisticsLog-likelihood maximization (MLE) converts the product of many probabilities into a sum using ln: ln(∏ P(xᵢ)) = Σ ln(P(xᵢ)). This is computationally essential — multiplying many small probabilities causes numerical underflow; summing their logs does not.
GrowthContinuous exponential growth and decay models use e: P(t) = P₀·eᵏᵗ. This models population growth, compound interest (continuously compounded), viral spread, and radioactive decay. Solving for t requires ln.
DataLog transformation using ln is the standard way to handle right-skewed distributions (income, prices, counts). It normalizes data and stabilizes variance. ln(x+1) is used when data includes zeros (log1p transform).
Info TheoryWhen using natural log in the entropy formula, H(X) = −Σ p(x) · ln(p(x)), the result is measured in nats (natural units of information) rather than bits. This is the preferred form in ML theory.

Criterion 03
How This Works

Part 1 — Understanding e through continuous growth

Imagine a bank account earning 100% interest per year. If interest compounds once a year: you end with (1 + 1)¹ = $2. If it compounds monthly: (1 + 1/12)¹² ≈ $2.613. Daily: (1 + 1/365)³⁶⁵ ≈ $2.714. As compounding becomes continuous, the limit approaches exactly e ≈ 2.71828. This is where e comes from — it is the result of perfectly continuous growth.

Compounding formula: A = P·(1 + r/n)^(nt) As n → ∞ (continuous compounding): A = P · eʳᵗ e is the natural base of continuous growth.

Part 2 — All 8 Essential Properties of ln

Domain & Range
Domain: x > 0 only
Range: (−∞, +∞)
ln is undefined for x ≤ 0. As x → 0⁺, ln(x) → −∞. As x → ∞, ln(x) → +∞ (slowly).
Identity Values
ln(1) = 0
ln(e) = 1
ln(eⁿ) = n
These three are the most used in simplification. Memorize them cold.
Inverse Relationship
e^(ln x) = x
ln(eˣ) = x
ln and eˣ cancel each other — they are perfect inverses. Used constantly in solving equations.
Product Law
ln(ab) = ln(a) + ln(b)
Turns multiplication into addition. Foundation of log-likelihood in statistics.
Quotient Law
ln(a/b) = ln(a) − ln(b)
Turns division into subtraction. Used in log-odds and KL divergence.
Power Law
ln(xⁿ) = n · ln(x)
Pulls exponent out as a multiplier. Critical for differentiating and simplifying ML loss functions.
Derivative
d/dx [ln(x)] = 1/x
d/dx [eˣ] = eˣ
These are among the most important derivatives in all of calculus. eˣ is the only function that is its own derivative.
Change of Base
log_b(x) = ln(x) / ln(b)
Any log in any base can be computed using ln. Python's math.log(x, b) uses this internally.

Part 3 — Solving equations with ln and e

Two main types of equations appear in analytics:

A
Exponential equation (unknown in exponent) → take ln of both sides to bring the exponent down using the Power Law, then solve.
B
Logarithmic equation (unknown inside ln) → isolate the ln, then raise e to both sides to remove the ln using the inverse law.
Type A — Exponential equation: e^(3x) = 45 ln(e^(3x)) = ln(45) ← take ln of both sides 3x = ln(45) ← ln and e cancel (inverse law) x = ln(45)/3 ≈ 3.807/3 ≈ 1.269 Type B — Logarithmic equation: ln(2x − 1) = 4 e^(ln(2x−1)) = e⁴ ← raise e to both sides 2x − 1 = e⁴ ← e and ln cancel (inverse law) 2x = e⁴ + 1 x = (e⁴ + 1)/2 ≈ (54.598 + 1)/2 ≈ 27.799

Criterion 04
Sample Problems
Sample A — Evaluating ln expressions
Evaluate each using properties of ln: a) ln(e⁷) = 7 ← Power Law: ln(eⁿ) = n b) e^(ln 15) = 15 ← Inverse Law: e^(ln x) = x c) ln(1) = 0 ← Identity: ln(1) = 0 d) ln(e⁻⁴) = −4 ← Power Law e) ln(√e) = ln(e^(1/2)) = 1/2 · ln(e) = 1/2 · 1 = 0.5 ← Power Law + Identity
✔ These evaluate without a calculator — they use only the fundamental properties. Recognizing these on sight is an essential fluency skill.
Sample B — Simplifying with ln Laws
Simplify: ln(x²y) − ln(√x) + 3·ln(z) Step 1 — Expand each term: ln(x²y) = ln(x²) + ln(y) = 2·ln(x) + ln(y) ← Product + Power ln(√x) = ln(x^(1/2)) = (1/2)·ln(x) ← Power Law 3·ln(z) = ln(z³) ← Power Law (reverse) Step 2 — Combine: [2·ln(x) + ln(y)] − (1/2)·ln(x) + 3·ln(z) = (2 − 1/2)·ln(x) + ln(y) + 3·ln(z) = (3/2)·ln(x) + ln(y) + 3·ln(z) Step 3 — Condense (optional): = ln(x^(3/2)) + ln(y) + ln(z³) = ln( x^(3/2) · y · z³ )
✔ Expand fully first, collect like-ln terms, then condense if needed. Treat ln(x), ln(y), ln(z) as different "variables" when combining.
Sample C — Solving an Exponential Equation
A model's accuracy grows continuously: A(t) = 0.5 · e^(0.3t) Find t when accuracy reaches 0.95. 0.95 = 0.5 · e^(0.3t) Step 1 — Isolate eˣ: e^(0.3t) = 0.95 / 0.5 = 1.9 Step 2 — Take ln: ln(e^(0.3t)) = ln(1.9) Step 3 — Simplify: 0.3t = ln(1.9) Step 4 — Solve: t = ln(1.9) / 0.3 = 0.6418 / 0.3 t ≈ 2.14 time units Verify: 0.5 · e^(0.3 × 2.14) = 0.5 · e^0.642 ≈ 0.5 × 1.9 = 0.95 ✓
✔ Standard structure: isolate eˣ → take ln → use inverse law to remove e → solve linearly.
Sample D — Log-Likelihood Using ln (Statistics)
Observed data: x₁=2, x₂=3, x₃=5 from a Poisson distribution P(x;λ) P(x;λ) = e^(−λ) · λˣ / x! Likelihood L(λ) = P(2)·P(3)·P(5) ← product of three probabilities Log-likelihood ℓ(λ) = ln(L) = ln(P(2)) + ln(P(3)) + ln(P(5)) For each term: ln(P(x;λ)) = ln(e^(−λ) · λˣ / x!) = ln(e^(−λ)) + ln(λˣ) − ln(x!) ← Product + Quotient = −λ + x·ln(λ) − ln(x!) ← Power + Inverse Law Summing all three: ℓ(λ) = −3λ + (2+3+5)·ln(λ) − [ln(2!) + ln(3!) + ln(5!)] = −3λ + 10·ln(λ) − constant To find best λ: take derivative, set to 0 → λ̂ = 10/3 ≈ 3.33
✔ Every step of log-likelihood uses ln laws. The power law (x·ln(λ)) is the key transform that makes the derivative tractable.
Sample E — Sigmoid Function and Log-Odds (Logistic Regression)
Sigmoid function: σ(z) = 1 / (1 + e^(−z)) The log-odds (logit) is the inverse of sigmoid: If p = σ(z), then: logit(p) = ln( p / (1−p) ) = z Worked example — compute sigmoid and log-odds: z = 1.5 σ(1.5) = 1 / (1 + e^(−1.5)) = 1 / (1 + 0.2231) = 1 / 1.2231 ≈ 0.8176 (81.76% predicted probability of class 1) logit(0.8176) = ln(0.8176 / 0.1824) = ln(4.482) ≈ 1.5 ✓ (recovers original z)
✔ Logistic regression is entirely built on e and ln. The sigmoid uses eˣ; the log-odds uses ln. They are inverse functions — this is why logistic regression is called a "log" model.
Sample F — Continuous Compound Interest (Real World)
You invest $5,000 at 6% annual interest, continuously compounded. How long until it doubles? A = P · eʳᵗ 10,000 = 5,000 · e^(0.06t) ← A = 10,000 (double), P = 5,000 Step 1: 2 = e^(0.06t) Step 2: ln(2) = 0.06t Step 3: t = ln(2) / 0.06 = 0.6931 / 0.06 t ≈ 11.55 years Rule of 70 shortcut: ≈ 70 / interest rate% = 70/6 ≈ 11.67 years ✓
✔ ln(2) ≈ 0.693 is a value worth memorizing — it appears in doubling-time problems, half-life problems, and learning rate tuning.

Criterion 05
Common Challenges & How to Solve Them
Confusing ln(eˣ) = x with e^(ln x) = x — not recognizing that both apply, or mixing up which direction cancels what.
Fix: Both inverse laws are true and work in both directions: ln(eˣ) = x (ln undoes e), and e^(ln x) = x (e undoes ln). Think of them as a lock and key — each opens the other. When you see e inside an ln, or ln inside an e exponent, they cancel completely. Example: ln(e^(x+3)) = x+3. Example: e^(ln(2x)) = 2x. Practice recognizing these on sight before solving anything else.
Thinking ln(x + y) = ln(x) + ln(y) — the same critical error as with general logs, but even more common with ln.
Fix: Repeating from log laws because it is the most common error: the Product Law applies to multiplication ONLY — ln(x · y) = ln(x) + ln(y). Addition inside the ln cannot be split. ln(3 + 5) = ln(8) ≈ 2.08, NOT ln(3) + ln(5) = 1.099 + 1.609 = 2.708. These are different numbers. There is no ln law for addition or subtraction inside the argument.
Trying to evaluate ln of zero or a negative number — causes undefined errors in code and wrong answers on paper.
Fix: ln(x) is defined only for x > 0. ln(0) = −∞ (approaches, never equals). ln(negative) = undefined in real numbers. In data analytics: if your feature column has zeros, use np.log1p(x) which computes ln(x+1) — this is defined for x ≥ 0. If it has negatives, you need a different transform (e.g., signed log: sign(x)·ln(|x|+1)). Always check for zeros/negatives before applying log transforms.
Forgetting that Python's math.log(x) is the natural log (ln), not log base 10.
Fix: In Python, math.log(x) and np.log(x) both compute ln(x) — the natural log. For base-10, use math.log10(x) or np.log10(x). For base-2, use math.log2(x). For any custom base b: math.log(x, b). This trips up people coming from Excel, where LOG(x) defaults to base 10 and LN(x) is the natural log.
Misapplying the Power Law — writing n · ln(x) = ln(n · x) instead of ln(xⁿ).
Fix: The Power Law is: ln(xⁿ) = n · ln(x). The n becomes a multiplier OUTSIDE, and x stays as the argument with its exponent removed. It does NOT become ln(n·x). The difference: ln(x³) = 3·ln(x), but 3·ln(x) ≠ ln(3x). Example: ln(8) = ln(2³) = 3·ln(2) ≈ 3×0.693 = 2.079. But ln(3·2) = ln(6) ≈ 1.792. Completely different.
Not isolating the exponential term before taking ln — applying ln too early and creating a messier expression.
Fix: When solving e^(something) = value, always isolate the exponential part on one side FIRST, then take ln of both sides. If you have 3·e^(2x) + 5 = 20, first subtract 5 to get 3·e^(2x) = 15, then divide by 3 to get e^(2x) = 5, THEN take ln: 2x = ln(5). Taking ln before isolating gives ln(3·e^(2x)) which requires extra steps to simplify.
Numerical underflow — multiplying many small probabilities together instead of summing their logs.
Fix: In statistics and ML, probabilities are often very small numbers like 0.001, 0.003, etc. Multiplying 1,000 such values together produces a number so tiny the computer rounds it to zero (underflow). The solution is exactly what log-likelihood does: take ln of each probability and sum them. ln(0.001) = −6.9, and summing −6.9 × 1000 = −6900 is perfectly representable. This is not just a math trick — it is a numerical stability requirement in real implementations.
